01 Background

The FarmerChat SDK is a pnpm/Turborepo monorepo that ships Android libraries as two separate packages (packages/android-compose and packages/android-views) plus a demo application at apps/demo-android. Each package was originally configured as a standalone Gradle project; opening any of them in Android Studio as a multi-module project exposed several configuration gaps.

ℹ
Correct entry point for Android Studio: Open apps/demo-android/ — not the monorepo root. The demo app's settings.gradle.kts already wires the two library packages as sub-projects via relative projectDir paths.

02 Issues Found & Fixed

1
Incomplete version catalog — missing 10+ library/plugin entries
apps/demo-android/gradle/libs.versions.toml
Fixed

When Android Studio opens apps/demo-android as the root Gradle project, its version catalog (gradle/libs.versions.toml) becomes the only catalog for all three modules. The two sub-projects (:android-compose, :android-views) have their own standalone catalogs which are silently ignored in multi-project builds. Their libs.* references therefore resolved against the demo-android catalog — which was missing every entry the libraries needed.

Missing entries (before fix)
# Versions not present:
coroutines, navigation, junit5

# Libraries not present:
compose-material-icons       # android-compose → libs.compose.material.icons
lifecycle-viewmodel-compose  # android-compose
lifecycle-viewmodel          # android-views
lifecycle-runtime            # android-views
navigation-fragment          # android-views
navigation-ui                # android-views
coroutines-android           # both libraries
coroutines-test              # android-compose tests
junit5                       # both libraries

# Plugins not present:
android-library              # both library modules use alias(libs.plugins.android.library)

All entries were added to the demo-android catalog so the multi-module build resolves correctly.

2
Plugin classpath conflict — com.android.library "unknown version"
apps/demo-android/build.gradle.kts
Fixed

Error: "The request for this plugin could not be satisfied because the plugin is already on the classpath with an unknown version, so compatibility cannot be checked."

Both com.android.application and com.android.library live in the same AGP jar (com.android.tools.build:gradle:8.13.0). When the root project applies com.android.application, Gradle loads the entire AGP jar but only registers the version for that one plugin ID. Sub-projects then request com.android.library:8.13.0 — Gradle sees the class already on the classpath with no tracked version and throws the compatibility error.

Fix — declare android.library with apply false in root build.gradle.kts
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.android.library) apply false   // registers version; not applied to root
    alias(libs.plugins.kotlin.compose)
}

apply false tells Gradle to register the plugin and its version (8.13.0) in the registry without applying it to the root project. Sub-projects can then apply com.android.library with the known version, and the compatibility check passes.

3
JUnit 5 tests silently skipped — useJUnitPlatform() not configured
packages/android-compose/build.gradle.kts · packages/android-views/build.gradle.kts
Fixed

Both library modules declare testImplementation(libs.junit5) but had no instruction for Gradle to use the JUnit Platform engine. Without this, Gradle's default JUnit 4 runner discovers zero tests and reports success — silently discarding all JUnit 5 test cases.

Fix added to both library build.gradle.kts files
android {
    // ... existing config ...

    testOptions {
        unitTests.all {
            it.useJUnitPlatform()
        }
    }
}

gradle.properties missing Kotlin style and performance settings
apps/demo-android/gradle.properties
Fixed

The demo app's gradle.properties had only 3 entries. The two library packages both declare kotlin.code.style=official, org.gradle.parallel=true, and org.gradle.caching=true, which apply to all sub-projects only when set in the root project. Missing these caused slower builds and inconsistent IDE inspection.

6
Script compilation error — kotlin { } DSL unavailable in all three modules
apps/demo-android/build.gradle.kts · packages/android-compose/build.gradle.kts · packages/android-views/build.gradle.kts
Fixed

Errors reproduced in all three build files:

e: build.gradle.kts:26:5: (android-compose / android-views)
e: build.gradle.kts:36:5: (demo-android)
None of the following candidates is applicable:
  fun DependencyHandler.kotlin(module: String, ...): Any
  fun PluginDependenciesSpec.kotlin(module: String): PluginDependencySpec

Unresolved reference 'compilerOptions'
Unresolved reference 'jvmTarget'

All three build.gradle.kts files contained a kotlin { compilerOptions { jvmTarget.set(JVM_17) } } block inside android {}. This DSL extension is registered by the Kotlin Android Gradle Plugin (org.jetbrains.kotlin.android), which is not applied in this project — AGP's built-in Kotlin is used instead. AGP's built-in Kotlin does not expose the kotlin {} script extension, so the reference fails at script compilation time in every module.

The compileOptions block (already present in all three files) is the correct way to set the JVM target when using AGP's built-in Kotlin — AGP automatically syncs the Kotlin jvmTarget from sourceCompatibility / targetCompatibility. The redundant kotlin {} block was removed from all three files.

Before — in all three build.gradle.kts (caused compilation failure)
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }

    kotlin {
        compilerOptions {
            jvmTarget.set(org.jetbrains.kotlin.gradle.dsl.JvmTarget.JVM_17)
        }
    }   // ← removed from demo-android, android-compose, android-views
After — compileOptions is sufficient; AGP syncs Kotlin jvmTarget automatically
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }

AGP version incompatible with installed Android Studio (9.1.0 → 8.13.0)
apps/demo-android/gradle/libs.versions.toml · packages/android-compose/gradle/libs.versions.toml · packages/android-views/gradle/libs.versions.toml · CLAUDE.md
Fixed

Error from Android Studio: "The project is using an incompatible version (AGP 9.1.0) of the Android Gradle plugin. Latest supported version is AGP 8.13.0."

Android Studio has a built-in compatibility table that maps IDE versions to the maximum AGP they support. When the IDE sees a newer AGP in the version catalog, it blocks the sync entirely before Gradle even runs. The project was authored targeting AGP 9.1.0, but the installed Android Studio only supports up to AGP 8.13.0.

AGP 8.13.0 still includes built-in Kotlin support (introduced in AGP 8.5+), so no other plugin changes were needed — only the version number required updating across all three libs.versions.toml files that declare the agp version ref, plus CLAUDE.md to keep documentation accurate.

Fix — updated in all three version catalogs
[versions]
agp = "8.13.0"   # was "9.1.0" — downgraded to match installed Android Studio
...

ViewBinding Java files cannot find Kotlin classes — kotlin-android plugin missing from library modules
packages/android-views/build.gradle.kts · packages/android-compose/build.gradle.kts
Fixed

Errors in generated ViewBinding files:

FragmentChatBinding.java:24:  package org.digitalgreen.farmerchat.views.ui.views does not exist
FragmentChatBinding.java:25:  package org.digitalgreen.farmerchat.views.ui.views does not exist
FragmentChatBinding.java:41:  cannot find symbol class ConnectivityBannerView
FragmentChatBinding.java:50:  cannot find symbol class InputBarView

The Kotlin source files (ConnectivityBannerView.kt, InputBarView.kt) exist in org.digitalgreen.farmerchat.views.ui.views. The error is a compilation ordering failure: ViewBinding generates Java source files from XML layouts, and those Java files import the custom Kotlin view classes. For the Java compiler to resolve those imports, the Kotlin sources must already be compiled to .class files first.

With AGP 8.13.0, the built-in Kotlin support is not automatic for library modules without an explicit Kotlin plugin. android-views had zero Kotlin plugins — only android.library — so no Kotlin compilation task was registered. ViewBinding's Java processing therefore ran without Kotlin classes on the classpath. android-compose had kotlin.compose (the Compose compiler plugin) but that is a compiler extension, not a full Kotlin Gradle setup; without kotlin-android as the foundation, it is also incomplete.

Adding kotlin-android to both library modules registers the Kotlin compilation tasks, ensuring Kotlin sources are compiled before ViewBinding Java files are compiled. In AGP 8.x this plugin is supported and required; only AGP 9.0+ rejects it as redundant with its own built-in Kotlin.

Fix — android-views/build.gradle.kts
plugins {
    alias(libs.plugins.android.library)
    alias(libs.plugins.kotlin.android)   // registers Kotlin compilation tasks
    id("maven-publish")
}
Fix — android-compose/build.gradle.kts
plugins {
    alias(libs.plugins.android.library)
    alias(libs.plugins.kotlin.android)   // foundation for Kotlin + compose compiler
    alias(libs.plugins.kotlin.compose)
    id("maven-publish")
}
Fix — apps/demo-android/build.gradle.kts (root registration)
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.android.library)  apply false
    alias(libs.plugins.kotlin.android)   apply false   // registers version for sub-projects
    alias(libs.plugins.kotlin.compose)
}

With kotlin-android applied the Kotlin compiler defaults to JVM 21, while compileOptions targets JVM 17 — Gradle enforces both must match (see Issue #9 for the JVM alignment fix and DSL path taken). The kotlin-android entry was also added to all three libs.versions.toml files so both standalone and multi-project builds resolve the plugin correctly.

JVM target mismatch + kotlinOptions deprecated — Kotlin 21 vs Java 17
packages/android-compose/build.gradle.kts · packages/android-views/build.gradle.kts
Fixed

Error: "Inconsistent JVM-target compatibility detected for tasks 'compileDebugJavaWithJavac' (17) and 'compileDebugKotlin' (21)."

Once kotlin-android was applied (Issue #8), the Kotlin compiler took ownership of JVM target selection and defaulted to JVM 21 (Kotlin 2.3.x default). The Java compiler was still set to JVM 17 via compileOptions. Gradle's JVM target validation rejects any mismatch.

The first attempt used kotlinOptions { jvmTarget = "17" } inside android {} — this silenced the mismatch error but immediately triggered a deprecation warning: "'var jvmTarget: String' is deprecated. Please migrate to the compilerOptions DSL." The kotlinOptions API was retired in Kotlin 2.x in favour of a top-level kotlin { } extension block.

The correct fix is a top-level kotlin { compilerOptions { } } block placed outside android { }. This is the block provided by kotlin-android as a project-level extension — distinct from the failed inner-android attempt in Issue #6 (which had no Kotlin plugin at all).

Final fix — top-level block in both library build.gradle.kts files
android {
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17   // Java → JVM 17
        targetCompatibility = JavaVersion.VERSION_17
    }
    // no kotlinOptions here — deprecated
}

kotlin {                                              // top-level, NOT inside android {}
    compilerOptions {
        jvmTarget.set(org.jetbrains.kotlin.gradle.dsl.JvmTarget.JVM_17)   // Kotlin → JVM 17
    }
}

ClassNotFoundException: MainActivity — kotlin-android not applied to app module
apps/demo-android/build.gradle.kts
Fixed

Runtime crash on launch:

java.lang.RuntimeException: Unable to instantiate activity
  ComponentInfo{org.digitalgreen.farmerchat.demo/...MainActivity}
Caused by: java.lang.ClassNotFoundException:
  Didn't find class "org.digitalgreen.farmerchat.demo.MainActivity"

MainActivity.kt exists and is syntactically correct. The APK was installed and Android launched the activity declared in the manifest — but the class was absent from the DEX. This means Kotlin compilation never ran for the app module: all .kt files were silently skipped, producing an APK with only resources and manifest.

The root cause: in Issue #8, kotlin-android was added to the root build file with apply false — intentionally not applied to the root project, only registering its version for sub-projects. But the app module IS the root project. With apply false, no compileDebugKotlin task was registered for demo-android, so Gradle built the APK without compiling any Kotlin source.

Before (app module had no Kotlin compilation)
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.android.library)  apply false
    alias(libs.plugins.kotlin.android)   apply false   // ← NOT applied; no Kotlin tasks
    alias(libs.plugins.kotlin.compose)
}
After — kotlin-android applied (no apply false), kotlin.compilerOptions added
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.android.library)  apply false
    alias(libs.plugins.kotlin.android)                // ← applied; Kotlin tasks registered
    alias(libs.plugins.kotlin.compose)
}

// ... android { } block unchanged ...

kotlin {
    compilerOptions {
        jvmTarget.set(org.jetbrains.kotlin.gradle.dsl.JvmTarget.JVM_17)
    }
}

All three modules now have kotlin-android properly applied (not just registered), ensuring Kotlin compilation tasks exist for every module.

04 How to Open in Android Studio

  1. Install an Android Studio version that supports AGP 8.13.0 (check Help → About to confirm). Ensure Android SDK 36 is installed via SDK Manager.
  2. In Android Studio choose File → Open and select the folder:
    farmerchat-sdk/apps/demo-android/ — not the monorepo root.
  3. Android Studio will detect settings.gradle.kts and start Gradle sync. It automatically creates local.properties with your SDK path.
  4. Wait for sync to complete. The project tree will show three modules: :app (demo-android), :android-compose, :android-views.
  5. Select a device/emulator (API 26+ required, matching minSdk = 26) and press Run.
⚠
API key: The app builds with a placeholder key (demo-key) by default. To use a real FarmerChat API key, add FC_API_KEY=your-key to apps/demo-android/local.properties before syncing.

05 Multi-Project Architecture (Reference)

📐
Each Android package is a valid standalone Gradle project AND a sub-project of demo-android.
Standalone builds: run ./gradlew build inside each package directory.
Multi-module (Android Studio): open apps/demo-android/ — includes both packages via projectDir paths. 
// apps/demo-android/settings.gradle.kts
include(":android-compose")
include(":android-views")

project(":android-compose").projectDir = file("../../packages/android-compose")
project(":android-views").projectDir   = file("../../packages/android-views")

When sub-projects are included this way, their own settings.gradle.kts and gradle/libs.versions.toml are ignored — only the root's (demo-android) catalog applies. This is why the catalog consolidation in Issue #1 was necessary.
