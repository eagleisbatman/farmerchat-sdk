#1
Missing Bundle Identifier — BKSHIDEvent crash on Simulator launch
apps/demo-ios · iOS Simulator
Error
failure in void __BKSHIDEvent__BUNDLE_IDENTIFIER_FOR_CURRENT_PROCESS_IS_NIL__
missing bundleID for main bundle NSBundle
<.../Build/Products/Debug-iphonesimulator> (loaded): {}
Root Cause
The project was originally a Swift Package (opened via Package.swift) using an executableTarget. SPM executable targets are designed for command-line tools — they have no CFBundleIdentifier, no Info.plist, and no concept of an iOS app bundle. When the iOS Simulator tries to launch any binary, it requires a bundle identifier to register the app with SpringBoard. Without one, it crashes immediately with the BKSHIDEvent assertion.
Fix — Created FarmerChatDemo.xcodeproj
Replaced the Swift Package workflow with a proper Xcode project containing a native iOS App target. The target sets:
  • PRODUCT_BUNDLE_IDENTIFIER = org.digitalgreen.farmerchat.demo
  • GENERATE_INFOPLIST_FILE = YES (Xcode auto-builds Info.plist)
  • IPHONEOS_DEPLOYMENT_TARGET = 16.0
  • SWIFT_VERSION = 5.9
  • Local SPM package references to ios-swiftui and ios-uikit

#2
Missing package product 'FarmerChatUIKit'
apps/demo-ios/FarmerChatDemo.xcodeproj · Xcode Package Resolution
Error
.../apps/demo-ios/FarmerChatDemo.xcodeproj
Missing package product 'FarmerChatUIKit'
Root Cause
After the xcodeproj was created, the original Package.swift (an SPM manifest) remained in the same directory (apps/demo-ios/). When Xcode opens a folder that contains both a .xcodeproj and a Package.swift, it can enter an ambiguous state — attempting to resolve package dependencies using the manifest's own dependency graph rather than the xcodeproj's XCLocalSwiftPackageReference entries. This mismatch prevents Xcode from registering the products of the local packages, producing the "Missing package product" error.
Fix — Disabled conflicting Package.swift
Renamed Package.swift → Package.swift.disabled in apps/demo-ios/. The xcodeproj now serves as the sole project definition for the demo app. The local iOS SDK libraries (packages/ios-swiftui, packages/ios-uikit) each retain their own Package.swift for building as standalone libraries, which is unaffected.

Why the original Package.swift was wrong
The original Package.swift used executableTarget, which is for CLI tools, not iOS apps. It was a workaround that caused two problems: the bundle ID crash (Issue #1) and, after the xcodeproj was added, the package conflict (Issue #2). The proper solution is the xcodeproj.

How to Open & Run in Xcode
  • 1
    Close the current tab in Xcode if apps/demo-ios is still open as a Swift Package.
  • 2
    File → Open, navigate to farmerchat-sdk/apps/demo-ios/FarmerChatDemo.xcodeproj and open it.
  • 3
    Wait for Xcode to finish "Resolving Package Versions" (progress bar at the top). If it doesn't auto-start, go to File → Packages → Resolve Package Versions.
  • 4
    If any package still shows as missing after resolution, use File → Add Package Dependencies…, click "Add Local…", and navigate to packages/ios-swiftui then packages/ios-uikit separately. Add FarmerChatSwiftUI and FarmerChatUIKit products to the FarmerChatDemo target.
  • 5
    Set the API key: Edit Scheme (⌘ <) → Run → Arguments → add Environment Variable FC_API_KEY = your-real-key.
  • 6
    Select an iPhone 15 (iOS 16+) Simulator and press ▶ Run.
ℹ️
The demo app uses ProcessInfo.processInfo.environment["FC_API_KEY"] as the API key. If not set via the scheme, it falls back to "demo-key". The app will launch but API calls will return 401 until a real key is provided.
⚠️
Do not re-add Package.swift to apps/demo-ios/. The xcodeproj is now the authoritative project file. The disabled file is kept only for historical reference. Restoring it will reintroduce the "Missing package product" conflict.
